# Amazon SKU Profit Summary

## 适用范围

使用本 Skill 的场景：

- 用户要求计算、核对或解释 Amazon SKU/MSKU 利润。
- 用户提供 Amazon transaction 风格主交易报告，以及 SKU 映射、订单、广告、仓储、移除、入库、库存等辅助文件。
- 用户要求判断 MSKU、FNSKU、ASIN、空白 SKU、异常费用或费用池如何归属。

协同边界：

- 库存分类账数量核算由 `amazon-inventory-ledger-summary` 负责。
- 已有利润结果后的经营诊断由 `amazon-sku-profit-health-monitor` 负责。

## 核心执行原则

主交易报告是 SKU 利润核算的唯一金额真相。

主交易报告代表老板实际到账和实际被 Amazon 扣款的钱。辅助文件只能用于给主交易报告源行寻找直接 MSKU 证据、比例证据、冲突证据或未解决原因。

总账闭环是基本门禁。除非存在明确的数据错误、异常、排除依据或已披露的未解决原因，SKU 利润汇总必须把主交易报告中属于利润口径的全部资金流纳入闭环。

```text
总毛利 = 交易报告全部利润口径收入 - 交易报告全部利润口径支出
```

未能进入 SKU/MSKU 明细的金额，必须进入店铺维度、费用池、未计入利润项目或行级 evidence 披露；不得因为无法归因到 SKU 就让交易报告真实金额从利润核算中消失。

最终产物解析门禁：生成或更新任何用户可见 `.xlsx`、`.csv` 或其他表格产物后，必须先解析实际输出文件，检查 sheet、表头、行数、关键列、合计行、公式结果、状态列和异常提示均无误。未执行解析或解析发现错误时，不得向用户输出最终文件路径，不得声明任务完成。

固定处理顺序：

第一步，逐行读取主交易报告源行，并生成稳定 `source_row_id`。

第二步，按 `sku` 列分流：正常 seller SKU/MSKU 进入正常交易规则；FNSKU 或 Liquidations 进入映射规则；空白 SKU 进入费用池和证据穷举规则。

第三步，对空白 SKU 行按 `type -> description -> order id -> 映射表/辅助文件` 顺序寻找证据。

第四步，证据成立时才归因或分摊；证据不成立时记录未解决、缺少证据、缺少关联、映射冲突或排除原因。

不可违背：

- 辅助文件不得反向新增或修改主交易报告归因。
- 辅助文件不得新增主交易报告不存在的收入或费用。
- 主交易报告源行缺少可审计关联时，不得用辅助明细中的 MSKU 反向写入计算结果。
- 为了让金额闭合而吸收无关联金额，属于错误归因。
- 关联证据优先于金额闭合；对不上就是对不上，必须披露。

## 多币种核算原则

第一步，保留主交易报告原始金额和原始币种。汇率换算不能改写源行金额，也不能替代 `anchor_total`。

第二步，确定目标币种。用户已指定时按用户指定；未指定时优先使用 Settlement report 的结算/到账币种；仍无法确定时向用户确认。

第三步，按优先级选择汇率：Settlement report 结算汇率 > 用户或财务确认汇率 > `exchange_rate_lookup`（汇率查询 / Rate Exchange 工具）取得的外部市场参考汇率。市场参考汇率只能用于参考或降级测算，不能自动作为正式记账汇率。

第四步，所有跨币种金额都必须写入 `currency_conversion_evidence`，记录原始币种、目标币种、原始金额、折算金额、汇率、汇率日期、汇率来源、是否财务确认和适用范围。

第五步，有 Settlement report 且其中存在适用结算汇率时，必须优先使用；未使用时必须说明原因，否则不得声明正式核算完成。

## 利润核算结论的业务化表达四方视角

当需要解释 SKU/MSKU/FNSKU/ASIN 利润核算结果、费用归因、毛利率、净利率、广告费、FBA 费用、退款、仓储费、头程或其他费用变化时，最终面向用户的说明必须包含“四方视角”。核算数据仍以主交易报告和已验证辅助证据为准；四方视角只负责表达，不得改写公式、归因或金额。

1. 财务语言：用专业财务口径说明核算结论，必须带具体数据、期间、币种、字段口径、金额和比例，例如销售额、毛利、净利、毛利率、净利率、费用率或同比/环比变化。
2. 业务语言：把同一结论翻译成老板或业务人员容易理解的经营含义，必须保留关键数据。例如把“净利率 X%，同比下降 2 个百分点”表达为“每卖出 100 元商品，今年只留下 X 元净利润，比去年少 2 元”。
3. 进一步业务解释：说明导致变化的直接费用项、关联因子和可能的根本原因；区分已由源行和证据确认的原因、按规则分摊得到的原因、仍 unresolved 的原因。
4. 行动建议：只基于已验证事实提出具体动作，例如控投广告、复查退款原因、优化 FBA/仓储结构、调整售价、优化头程渠道或重新议价物流商；建议必须说明适用 SKU/品线/站点和预期影响，不得泛泛而谈。

## 脚本粒度约束

一个脚本只处理一个字段、一个费用池或一个明确归因层。

允许公共预处理脚本做读取、表头识别、字段清洗、`source_row_id` 生成、初始分层、映射索引构建或 workbook 读回校验。公共预处理脚本不得顺手计算多个利润字段。

每个计算脚本必须声明：

```text
目标字段或费用池
输入源行范围
使用公式
归因证据
输出文件
校验方式
```

每个脚本执行完成后，必须先验证当前脚本结果，再写或执行下一个脚本。

绝对不允许一个脚本计算大量任务（非原子任务）！

## 校验底线

每次计算都必须从主交易报告真实源行反向验证：

- 应处理的源交易行是否全部处理。
- 不应处理的源交易行是否没有越界吸收。
- 已归因到 MSKU 的金额是否能回到主交易报告源行和直接证据。
- 标记为 unresolved 的源交易行是否已经按当前节点允许路径尝试完毕。
- 本应 unresolved 的源交易行是否被错误并入 MSKU 汇总。
- 辅助文件与主交易报告不一致时，是否以主交易报告为准并披露差异。
